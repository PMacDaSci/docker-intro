Episode 1: Introducing Containers

Questions

  • What are containers, and why might they be useful to me?

Objectives

  • Show how software depending on other software leads to configuration management problems.
  • Identify the problems that software installation can pose for research.
  • Explain the advantages of containerization.
  • Explain how using containers can solve software configuration problems

Scientific Software Challenges

What’s Your Experience?

  • Think about software frustrations in your research workflow.
  • Compare with a neighbour and list common pain points.
  • Focus on installation, versions, portability, and reproducibility.

Scientific Software Challenges

Common Symptoms:

  • software is unavailable on your preferred OS;
  • installation pulls in many nested dependencies;
  • only some version combinations work;
  • colleagues get different results on different systems;
  • reproducing an older setup becomes difficult.

Why This Happens:

  • research software depends on many layers;
  • one missing or incompatible piece can break the stack;
  • this situation is often called dependency hell.

What Is a Software Dependency?

  • A dependency is software that another tool needs in order to work.
  • Example: an application may depend on a library to talk to a database.

Why This Matters for Research

  • you may be unable to use a required tool at all;
  • results become harder to reproduce with confidence;
  • moving work to new systems or collaborators becomes difficult;
  • others may be unable to validate or extend your work.

Containers help by packaging software, dependencies, and access to resources in a more consistent, reproducible way.

What Is a Container?

Think of a container as a self-contained working environment inside your existing computer:

  • isolated from your main software setup;
  • cheaper and lighter than a separate machine;
  • portable across systems with the same container engine.

More concretely, Docker Inc use the following definition of a container:

A container is a standard unit of software that packages up code and all its dependencies so the application runs reliably from one computing environment to another.

https://www.docker.com/resources/what-container/

The shipping-container analogy is useful:

  • containers keep related things together;
  • they standardize transport and reuse;
  • the same container can be run on any compatible host system.

Virtualization

Virtualization

Virtualization means running an isolated computing environment on a main (host) computer.

  • Containers are an example of virtualization
  • Virtual machines (VMs) include a full operating system and are heavier.
  • Containers are lighter and usually share the host’s Linux kernel.
  • Containers still provide an isolated filesystem and runtime for applications.

What Is Docker?

Docker is a platform for building and running containers.

Container Images

Key distinction:

  • a container image is the recipe or template;
  • a container is the running instance created from that image;
  • images persist, containers are usually transient.

The cookie-cutter analogy works well: one cutter creates many cookies of the same shape.

Putting the Pieces Together

Containers help because the environment becomes explicit, portable, and easier to recreate.

Benefits:

  • documentation – there is a clear record of what software and software dependencies were used, from bottom to top.
  • portability – the container can be used on any computer that has Docker installed – it doesn’t matter whether the computer is Mac, Windows or Linux-based.
  • reproducibility – you can use the exact same software and environment on your computer and on other resources (like a large-scale computing cluster).
  • configurability – containers can be sized to take advantage of more resources (memory, CPU, etc.) on large systems (clusters) or less, depending on the circumstances.

Use Cases for Containers

  • Run a specific tool locally without a difficult install.
  • Test software while avoiding dependency hell.
  • Build an image from a Dockerfile for a defined research setup.
  • Share that image with collaborators or on cloud and HPC systems.
  • Archive images so you can rerun work with the same software later.

Docker Command Line Interface (CLI) Syntax

Docker Command Line Interface (CLI) Syntax

This lesson uses the newer grouped Docker CLI syntax introduced with the Docker Engine version 1.13:

  • docker [command] [subcommand] [options]
  • common groups include image and container
  • this form is clearer than the older flat command style

Comparing the help outputs shows that Docker supports both old and newer CLI forms.

  • Older form: docker run
  • Newer grouped form: docker container run

The grouped form is more descriptive and is the one used in this workshop.

Running the hello-world Container

docker container run does three things:

  1. Starts a running container from the chosen image. Think of this as the “alive” or “inflated” version of the container.
  2. Executes the image’s default command. This could be as simple as printing a message (as above) or running a whole analysis pipeline!
  3. Once the default action is complete, the container stops running (or exits). The container image is still there, but nothing is actively running.

For hello-world, the default command prints this diagnostic message and exits.

Using docker container run to Get the Image

  • You can skip docker image pull.
  • If the image is missing, docker container run will pull it automatically and then run it.

Run the Alpine Docker Container

Running Containers Interactively

To run interactively,

  • add the interactive flags -i and -t (usually combined as -it)
  • provide a shell (bash, sh, etc.) as the command
  • For alpine, use sh because bash is not included.
docker container run -it alpine sh

Technically…

  • -i keeps standard input open.
  • -t allocates a text terminal (pseudo-TTY).
  • In practice, -it is the usual combination for interactive shells.

Running Containers Interactively

Your prompt should change to look like this:

/ #

You’re now inside the running container! Try these commands:

  • pwd

  • ls

  • whoami

  • echo $PATH

  • cat /etc/os-release

  • These commands run inside the container, not on your host computer.

  • The output describes the container environment.

  • When finished, exit the container with exit.

/ # exit

docker container run supports many options beyond -it. Here are two common ones:

Episode 4: Cleaning Up Containers

Questions

Objectives

Removing Images

Containers and images can accumulate and consume disk space over time.

To remove an image, identify it first, usually by Image ID:

docker image ls
REPOSITORY       TAG         IMAGE ID       CREATED          SIZE
hello-world      latest      fce289e99eb9   15 months ago    1.84kB

Remove by image ID:

docker image rm fce289e99eb9

Or remove by image name:

docker image rm hello-world

You may see this error:

Error response from daemon: conflict: unable to remove repository reference "hello-world" (must force) - 
container e7d3b76b00f4 is using its referenced image fce289e99eb9

Why this happens:

  • A container (running or stopped) is still linked to that image.
  • Remove relevant containers first, then remove the image.

What Containers Are Running?

Use docker container commands to inspect container state.

List currently running containers:

docker container ls
CONTAINER ID        IMAGE               COMMAND             CREATED             STATUS              PORTS               NAMES

No rows means no containers are currently running.

docker ps

  • docker ps is equivalent to docker container ls.
  • The name comes from the Unix ps command which describes running processes.

What Containers Have Run Recently?

To include stopped containers, add --all (or -a):

docker container ls --all
CONTAINER ID        IMAGE               COMMAND             CREATED             STATUS                     PORTS               NAMES
9c698655416a        hello-world         "/hello"            2 minutes ago       Exited (0) 2 minutes ago                       zen_dubinsky
6dd822cf6ca9        hello-world         "/hello"            3 minutes ago       Exited (0) 3 minutes ago                       eager_engelbart

Keeping It Clean

  • Docker keeps stopped containers unless you remove them.
  • Add --rm to delete the container automatically when it exits.
  • Do not use --rm if you need to inspect or reconnect to that container later.

How Do I Remove an Exited Container?

Delete one exited container by ID:

  • Replace with the container ID you want to remove.
  • Docker echoes the ID on success.
docker container rm 9c698655416a
9c698655416a

Use docker container prune to delete all stopped containers.

  • It does not take a container ID.
  • Deletion is permanent.
  • Avoid this command if you may need to reconnect to stopped containers.
  • Docker asks for confirmation and then lists removed container IDs.
docker container prune
WARNING! This will remove all stopped containers.
Are you sure you want to continue? [y/N] y
Deleted Containers:

Removing Images, for Real This Time

After removing dependent containers, retry image removal:

docker image rm hello-world
Untagged: hello-world:latest
Untagged: hello-world@sha256:5f179596a7335398b805f036f7e8561b6f0e32cd30a32f5e19d17a3cda6cc33d
Deleted: sha256:fce289e99eb9bca977dae136fbe2a82b6b7d4c372474c9235adc1741675f587e
Deleted: sha256:af0b15c8625bb1938f1d7b17081031f649fd14e6b233688eea3c5483994a66a3

Why multiple lines appear:

  • Docker images are made of layered filesystems.
  • Removing one image may untag and/or delete several underlying layers.
  • Shared layers are kept until no image depends on them.

Episode 5: Finding Containers on Docker Hub

Introducing the Docker Hub

Docker Hub is an online registry of container images.

  • Many images are public.
  • Many are maintained by software developers.
  • Some are official/endorsed images with stronger trust signals.
  • Trusted images are more likely to be maintained, stable, and secure.

Docker Can Be Used without Connecting to Docker Hub

  • Docker Hub is convenient, but not required.
  • Some organizations use private or offline container infrastructure instead.

Exploring an Example Docker Hub Page

Use the official Python image page as an example: https://hub.docker.com/_/python

What to look for on a Docker Hub page:

  • Top-left: image name, short description, popularity, endorsements.

  • Top-right: pull command.

  • Main body sections typically include:

    • Which tags (i.e., container image versions) are supported;
    • Summary information about where to get help, which computer architectures are supported, etc.;
    • A longer description of the container image;
    • Examples of how to use the container image; and
    • The license that applies.

The “How to use this image” section is usually the best starting point.

Exploring Container Image Versions

A single Docker Hub repository can provide many image versions.

  • Versions are represented by tags.
  • Use : to specify a tag:
CONTAINER_IMAGE_NAME:TAG

To download Python 3.8 container image:

docker image pull python:3.8

To download Python 3.6 container image:

docker image pull python:3.6

If no tag is provided, Docker uses the default tag - latest.

Images from Others

Not all images are maintained by Docker.

  • Many are published by individuals or organizations.
  • Your own uploaded images follow this same model.
  • Examples include ContinuumIO (who develop the Anaconda Python environment) and rocker (group that builds community R container images).

Common naming pattern for owner-managed images:

OWNER/CONTAINER_IMAGE_NAME:TAG

Repositories

  • A Docker Hub page represents a repository.
  • A tag selects a specific version from that repository.

So a more precise naming pattern is:

OWNER/REPOSITORY:TAG

  • There is a Dockerfile or other listing of what has been installed to the container image.
  • The container image page has documentation on how to use the container image.

Avoid images that are stale, poorly documented, or from unclear sources.

  • They are harder to trust.
  • They reduce reproducibility and reliability for research workflows.

Put Installation Instructions in a Dockerfile

A Dockerfile is a plain-text recipe for building container images.

Inspect the starter Dockerfile:

cd ~/Desktop/docker-intro/basic
cat Dockerfile
FROM <EXISTING IMAGE>
RUN <INSTALL CMDS FROM SHELL>
CMD <CMD TO RUN BY DEFAULT>

What each instruction means:

  • FROM: which container image we’re starting with. It is the “base” container image we are going to start from.

  • RUN: installation commands we want to run. These are the same commands that we used interactively above.

  • CMD: the default command we want a container based on this container image to run, if no other command is provided.

    • It is recommended to provide CMD in exec-form (see the CMD section of the Dockerfile documentation for more details).
    • It is written as a list which contains the executable to run as its first element, optionally followed by any arguments as subsequent elements.
    • The list is enclosed in square brackets ([]) and its elements are double-quoted (") strings which are separated by commas.
    • For example, CMD ["ls", "-lF", "--color", "/etc"] would translate to ls -lF --color /etc.

Shell-Form and Exec-Form for CMD

Another way to specify the parameter for the CMD instruction is the shell-form. Here you type the command as you would call it from the command line. Docker then silently runs this command in the image’s standard shell. CMD cat /etc/passwd is equivalent to CMD ["/bin/sh", "-c", "cat /etc/passwd"]. We recommend to prefer the more explicit exec-form because we will be able to create more flexible container image command options and make sure complex commands are unambiguous in this format.

Running Containers

What command would we use to run Python from the alpine-python container?

Running Containers Solution

Solution

We can run a container from the alpine-python container image using:

docker container run alice/alpine-python

What happens? Since the Dockerfile that we built this container image from had a CMD entry that specified ["python3", "--version"], running the above command simply starts a container from the image, runs the python3 --version command and exits. You should have seen the installed version of Python printed to the terminal.

Instead, if we want to run an interactive Python terminal, we can use docker container run to override the default run command embedded within the container image. So we could run:

docker container run -it alice/alpine-python python3

The -it tells Docker to set up and interactive terminal connection to the running container, and then we’re telling Docker to run the python3 command inside the container which gives us an interactive Python interpreter prompt. (type exit() to exit!)

If we try running the container and Python script, what happens?

docker container run alice/alpine-python python3 sum.py
python3: can't open file '//sum.py': [Errno 2] No such file or directory

No Such File or Directory

No Such File or Directory

What does the error message mean? Why might the Python inside the container not be able to find or open our script?

Why the Error Happens:

  • Container filesystems are isolated from the host.
  • The container cannot see files on your computer unless you expose them.
  • sum.py is on the host, not inside the container.

Fix: create a mount from host to container.

  • ${PWD} expands to your current host directory.
  • mount option:
`--mount type=bind,source=${PWD},target=/temp`

Interpretation:

make my current working directory (on the host computer) – the source – visible within the container that is about to be started, and inside this container, name the directory /temp – the target.

Types of Mounts

  • This example uses a bind mount.
  • Other mount types include volume and tmpfs.
  • We do not cover them here, but Docker documents them in more detail.

Accessing Mounted Files in the Container

Let’s try running the command now:

docker container run --mount type=bind,source=${PWD},target=/temp alice/alpine-python python3 sum.py

But we get the same error!

python3: can't open file '//sum.py': [Errno 2] No such file or directory

The missing detail: use the container path to the script.

  • sum.py is visible at /temp/sum.py inside the container.
  • so run Python with that full path:
docker container run --mount type=bind,source=${PWD},target=/temp alice/alpine-python python3 /temp/sum.py

Files written in /temp from inside the container are written to your host directory and remain after the container exits.

Other Commonly Used Docker Run Flags

Other Commonly Used Docker Run Flags

Two useful docker run flags are -w and -u.

The --workdir/-w flag sets the working directory a.k.a. runs the command being executed inside the directory specified. For example, the following code would run the pwd command in a container started from the latest ubuntu image in the /home/alice directory and print /home/alice. If the directory doesn’t exist in the image it will create it.

docker container run -w /home/alice/ ubuntu pwd

The --user/-u flag lets you specify the username you would like to run the container as. This is helpful if you’d like to write files to a mounted folder and not write them as root but rather your own user identity and group. A common example of the -u flag is --user $(id -u):$(id -g) which will fetch the current user’s ID and group and run the container as that user.

The Importance of Command Order in a Dockerfile

The Importance of Command Order in a Dockerfile

When you run docker image build it executes the build in the order specified in the Dockerfile. This order is important for rebuilding and you typically will want to put your RUN commands before your COPY commands.

Docker builds the layers of commands in order. This becomes important when you need to rebuild container images. If you change layers later in the Dockerfile and rebuild the container image, Docker doesn’t need to rebuild the earlier layers but will instead used a stored (called “cached”) version of those layers.

For example, in an instance where you wanted to copy multiply.py into the container image instead of sum.py. If the COPY line came before the RUN line, it would need to rebuild the whole image. If the COPY line came second then it would use the cached RUN layer from the previous build and then only rebuild the COPY layer.

Copying Alternatives

Copying Alternatives

Another trick for getting your own files into a container image is by using the RUN keyword and downloading the files from the internet. For example, if your code is in a GitHub repository, you could include this statement in your Dockerfile to download the latest version every time you build the container image:

RUN git clone https://github.com/alice/mycode

Similarly, the wget command can be used to download any file publicly available on the internet:

RUN wget ftp://ftp.ncbi.nlm.nih.gov/blast/executables/blast+/2.10.0/ncbi-blast-2.10.0+-x64-linux.tar.gz

Note that the above RUN examples depend on commands (git and wget respectively) that must be available within your container: Linux distributions such as Alpine may require you to install such commands before using them within RUN statements.

More Fancy Dockerfile Options

You can make the image more automatic with CMD and ENTRYPOINT patterns.

Make the sum.py Script Run Automatically

FROM alpine
RUN apk add --update python3 py3-pip python3-dev
COPY sum.py /home

# Run the sum.py script as the default command
CMD ["python3", "/home/sum.py"]

Build and test it:

docker image build -t alpine-sum:v1 .
docker container run alpine-sum:v1

Doesn’t work: Running with no arguments works (e.g. sum = 0). Passing arguments this way fails:

docker container run alpine-sum:v1 10 11 12

results in

docker: Error response from daemon: OCI runtime create failed:
container_linux.go:349: starting container process caused "exec:
\"10\": executable file not found in $PATH": unknown.

Reason: 10 11 12 is interpreted as a replacement command, not arguments to the existing CMD.

Make the sum.py Script Run Automatically

To always run a base command and still accept runtime arguments, use ENTRYPOINT plus CMD defaults.

FROM alpine

RUN apk add --update python3 py3-pip python3-dev
COPY sum.py /home

# Run the sum.py script as the default command and
# allow people to enter arguments for it
ENTRYPOINT ["python3", "/home/sum.py"]

# Give default arguments, in case none are supplied on
# the command-line
CMD ["10", "11"]

Build and test it:

docker image build -t alpine-sum:v2 .
# Most of the time you are interested in the sum of 10 and 11:
docker container run alpine-sum:v2
# Sometimes you have more challenging calculations to do:
docker container run alpine-sum:v2 12 13 14

Overriding the ENTRYPOINT

Overriding the ENTRYPOINT

  • Sometimes you want to bypass the image’s ENTRYPOINT.
  • For example, you may want an interactive shell instead of the default app.
docker container run -it alpine-sum:v2 /bin/sh

will yield

Please supply integer arguments

Override the ENTRYPOINT like this:

docker container run -it --entrypoint /bin/sh alpine-sum:v2

Add the sum.py script to the PATH so you can run it directly:

FROM alpine

RUN apk add --update python3 py3-pip python3-dev

COPY sum.py /home
# set script permissions
RUN chmod +x /home/sum.py
# add /home folder to the PATH
ENV PATH /home:$PATH

Build and test it:

docker image build -t alpine-sum:v3 .
docker container run alpine-sum:v3 sum.py 1 2 3 4

Sharing Images

Docker Hub is useful for day-to-day sharing and collaboration.

But it is not ideal for long-term research archiving because:

  • free accounts have a limit on how long a container image will be hosted if it is not updated
  • it does not support adding persistent identifiers to container images
  • it is easy to overwrite tagged container images with newer versions by mistake.

Archiving and Persistently Identifying Container Images Using Zenodo

For publications, archive images in an immutable, citable form. Zenodo is one option.

Zenodo accepts tar archives. Export an image with docker image save:

docker image save alice/alpine-python:v1 -o alpine-python.tar

If needed, compress before upload:

gzip alpine-python.tar

Note: Zenodo has upload size limits (for example, 50 GB per upload at the time of writing), so compression (gzip/zip) may be required.

Why Archive with Zenodo?

After creating the archive, deposit it on Zenodo to:

  • create a long-term archive snapshot that others can download and reuse
  • mint a persistent DOI (Digital Object Identifier) for citation in publications
  • support attribution and credit for your computational work

You will also provide metadata about the image and related work.

Zenodo is one option; some institutions provide equivalent archive/DOI services.

Reproducibility Good Practice

  • use images to capture the computational environment for your work
  • choose container granularity based on your workflow and domain norms
  • document what you did and why in both image docs and project docs
  • when publishing, archive the exact image and cite its DOI
  • use tags so older image versions remain accessible

Reproducibility Good Practice (Important Caveat)

  • a built and archived image gives a durable software snapshot
  • a Dockerfile is lightweight and useful for rebuilding later
  • if you only keep a Dockerfile, pin versions explicitly
  • unpinned dependencies can change and break reproducibility over time

Container Granularity

An important design choice is container granularity.

Two extremes are:

  • create one large image with all tools for the full workflow
  • create many small images, each for one command or workflow step

Most real workflows use a middle ground between these extremes.

This is not an exhaustive list but some of the advantages and disadvantages could be:

Single Large Container Image

  • Advantages:
    • Simpler to document
    • Full set of requirements packaged in one place
    • Potentially easier to maintain (though could be opposite if working with large, distributed group)
  • Disadvantages:
    • Could get very large in size, making it more difficult to distribute
      • Could use Docker multi-stage build to reduce size
    • May end up with same dependency issues within the container image from different software requirements
    • Potentially more complex to test
    • Less re-useable for different, but related, work

Multiple Smaller Container Images

  • Advantages:
    • Individual components can be re-used for different, but related, work
    • Individual parts are smaller in size making them easier to distribute
    • Avoid dependency issues between different pieces of software
    • Easier to test
  • Disadvantage:
    • More difficult to document
    • Potentially more difficult to maintain (though could be easier if working with large, distributed group)
    • May end up with dependency issues between component container images if they get out of sync
